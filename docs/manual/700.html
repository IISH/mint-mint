Persistent Identifiers

A user of a trusted system like the IALHI aggregator System expects to locate, retrieve and access digital objects and metadata via stable and persistent links on the web. This user requirement entails that the IALHI aggregator system must support PIDs.

IALHI aggregator starting point

Starting point of IALHI aggregator is that CPs should adopt PIDs because this is a best practice when disseminating metadata on the web and providing online access to digital collections.

PID definition

What makes an identifier a PID?

  • the PID should be a system independent and globally unique identifier on the Internet
  • the PID identifies a digital identity that is located on any system accessible via the Internet;
  • there is always an Internet-resolvable link (Resolve URL) to the location of the digital entity associated with the PID;
  • the content provider is responsible for binding the PID to the Resolve URL of the digital entity and for maintaining the binding.

Responsibility of the CP

CPs are thus responsible for assigning PIDs to their metadata and digital objects AND for assuring the binding of the PID to the resolve URL. CPs can carry out these tasks by use of a PID service.

What is a PID service?

  • A PID service is a tool (for content providers) to create, bind and maintain PIDs.
  • It is a service (to web users) that returns a resolve URL, which gives access to the digital entity.

CPs should therefore make use of a PID service and they are strongly advised to do so.

Syntax of a PID

When CPs register with a PID service, they will receive an institutional identifier that uniquely identifies their organization and this will be the prefix of their PID. For the PID-string to become an actionable URL, it needs to be preceded by the base-URL of the PID service.

The syntax of an actionable PID-string is as follows: [base URL PID service]/[prefix]/[suffix].

Example:

  • http://doi.acm.org/10.1145/602421.602422
  • [base URL PID service]: http://doi.acm.org => address of the PID service
  • /[prefix]: /10.1145 => prefix of the PID, identifying the CP
  • /[suffix]: /602421.602422 => suffix of the PID, generated by the PID service

We thus make a distinction between a PID, which is an identifier and an actionable PID-string, which is a URL.

When to use PIDs and when to use actionable PID-strings?

Because base URLs may change over time, it is a best practice to keep the PIDs and the base URL separated in the local metadata systems of the CPs. But as soon as the metadata is disseminated on the web to discovery services, they should contain actionable PID-strings.

In IALHI aggregator, the mapping sheets (mapping local metadata to domain profile metadata) provide the possibility to insert actionable PID-strings in the correct metadata fields.

PID requirements for CPs

For each metadata record submitted to IALHI aggregator, the CP should provide a PID-string for:

  1. the web page which displays the metadata record (landing page);
  2. the digital object described by the metadata record (if the object is online accessible);

PID Scenarios

From a CP perspective there are basically 2 PID scenarios:

  1. Scenario 1: CP implements PIDs
    • The CP chooses to implement PIDs and to adapt his local workflow to accommodate the creation, binding and maintenance of PIDS (this is the preferred and recommended scenario). In this scenario the CP can choose any PID service, including the IALHI aggregator PID service.
  2. Scenario 2: CP does not implement PIDs
    • The CP chooses not to implement PIDs and not to adapt his local workflow to accommodate PIDs (this is the “work-around scenario” which is strongly discouraged). In this scenario the IALHI aggregator system will assign and bind PIDs instead of the CP in order to ensure the quality of the metadata disseminated by IALHI aggregator.

Scenario 1: CP implements PIDs

Requirements

A CP following this PID scenario will need to accommodate PIDs in his local systems. In particular:

  1. the local metadata should have an element/field to accommodate the PID of the landing page;
  2. the local metadata should have an element/field to accommodate the PID of the digital object;

The CP should map these 2 elements/fields to the corresponding domain profile fields.

High-level workflow

  1. CP creates a PID for the digital object.
  2. CP stores digital object in repository (SOR or LOR) and gets a Resolve-URL in return.
  3. CP binds PID and Resolve-URL of the digital object in PID service.
  4. CP inserts PID of the digital object in corresponding metadata element.
  5. CP publishes metadata record on the web and gets a Resolve-URL of the landing-page in return.
  6. CP creates and binds PID with the Resolve-URL of the landing-page in PID service.
  7. CP inserts PID of the landing-page in corresponding metadata element.
  8. CP submits the metadata to the Aggregator.

For each step there are different variations/scenarios possible depending on which system is used, but on a high-level these variations are not important.

In particular: if a CP chooses the IALHI aggregator PID service, step 3 can be done by the SOR.

Scenario 2: CP does not implement PIDs

Requirements

A CP following this PID scenario will need to make use of local unique identifiers and URLs. In particular:

  1. the local metadata should have an element/field to accommodate the local id of the metadata record itself;
  2. the local metadata should have an element/field to accommodate the URL of the landing-page which displays the metadata record;
  3. the local metadata should have an element/field to accommodate the local id of the digital object which is described by the metadata;
  4. the local metadata should have an element/field to accommodate the URL of the digital object which is stored in a repository (if the CP uses a LOR);

The CP should map these 4 elements/fields to the corresponding domain profile fields. The Aggregator assigns and binds the corresponding PIDs (by use of the IALHI aggregator PID service) and maps them to the IALHI aggregator schema fields.

High-level workflow

  1. CP creates a LID for the digital object.
  2. CP stores digital object in a repository (SOR or LOR). If LOR: gets a URL in return.
  3. CP inserts local identifier (and if LOR: also the URL) of the digital object in the corresponding metadata element(s).
  4. CP publishes metadata record on the web and gets a URL of the landing-page in return.
  5. CP inserts local identifier of the metadata record and URL of the landing-page in the corresponding metadata elements.
  6. CP submits the metadata to the Aggregator.

For each step there are different variations/scenarios possible depending on which system is used, but on a high-level these variations are not important.

Detailed workflow for Choosing PID scenario 1

If you choose to implement PIDs, you need to take the following actions:

  1. Choose one of the existing PID services. Existing PID services from which you can choose are for example: the Handle System, ARK, DOI, the (click the following link:) IALHI aggregator PID Service, or any other third party PID service.
  2. Register your organization at the PID Service of your choice. You will receive an institutional identifier that uniquely identifies your organization with the PID Service.
  3. Provide this institutional identifier + the base-URL of your PID service to:
    1. the Aggregator, via the Aggregator user interface
    2. the SOR (if you use the SOR) via the SOR administration interface.
  4. Install your local resolver system. (OPTIONAL).

Detailed workflow for Choosing PID scenario 2

If you choose not to implement PIDs, you need to take the following actions:

  1. Define your local unique identifier (LID) for digital objects.(hyperlink to Best practices for LIDs)
  2. Define your local unique identifier (LID) for descriptive metadata records.(hyperlink to Best practices for LIDs)
  3. You need to (click the following link:) register your organization with the Handle system, in order to enable the Aggregator to assign PIDs to your digital entities by use of the IALHI aggregator PID service. You will receive an institutional identifier called 'naming authority number (NAN)' that uniquely identifies your organization with the Handle System.
  4. Provide this naming authority number + the base-URL of the IALHI aggregator PID service to:
    1. the Aggregator, via the Aggregator user interface
    2. the SOR (if you use the SOR) via the SOR administration interface.

Obligatory mappings related to the isShownAt URL

If your isShownAt PID is for example http://handle.net/12345/my-id​ and mapped so:

  • rdf:RDF/ore:Aggregation/edm:isShownAt/@rdf:resource​="http://handle.net/12345/my-id

Then also map to (note the concat of #cho and #aggregation at the end):

  • edm:ProvidedCHO@rdf:about="http://handle.net/12345/my-id#cho
  • ore:Aggregation/edm:aggregatedCHO/@rdf:resource="http://handle.net/12345/my-id​#cho
  • ore:Aggregation/@rdf:about="http://handle.net/12345/my-id​#aggregation
